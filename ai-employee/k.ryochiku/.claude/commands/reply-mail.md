# /reply-mail

受信トレイのメール（既読・未読両方）に対して、AIが返信文を作成します。
**3段階（返信必要・保留・返信不要）で自動分類**し、返信が必要なメールのみ即時処理します。
保留メールは今日は無視され、翌日以降に再度処理対象になります。

## 引数

- `$ARGUMENTS` — 以下のオプションを指定可能:
  - `--max=N` : 最大N件のメールを処理（デフォルト: 10）
  - `--from=ADDRESS` : 特定の送信者からのメールのみ対象
  - `--since=DATE` : 指定日以降のメール（例: 2026-03-01）
  - `--label=LABEL` : 特定のラベルのメールのみ対象
  - `--list` : 一覧表示のみ（返信作成しない）
  - `--all` : 分類をスキップして全件処理（デフォルトは返信必要のみ）
  - `--include-hold` : 保留中のメールも対象に含める
  - `--include-promo` : プロモーションメールも対象に含める（通常は自動除外）
  - `--include-social` : ソーシャル通知も対象に含める（通常は自動除外）

## 実行手順

### 1. 初期化

基本思考を読み込む:
```bash
cat /Users/kazui/scripts/ai-employee/_shared/philosophy.md
```

この内容を「メール返信時の考え方」として保持する。
文体には反映せず、提案内容・判断基準に反映する。

### 2. 未返信メールの取得

```bash
# 受信トレイのメールを検索（プロモーション・ソーシャル・保留を除外、メインのみ）
# ※既読・未読の両方を処理対象とする
gws gmail users messages list --params '{"userId": "me", "q": "in:inbox -label:_hold/today -category:promotions -category:social", "maxResults": 10}'
```

引数に応じてクエリを調整:
- `--from=ADDRESS` → `from:ADDRESS` を追加
- `--since=DATE` → `after:DATE` を追加
- `--label=LABEL` → `label:LABEL` を追加
- `--include-hold` → `-label:_hold/today` を除去して保留メールも含める
- `--include-promo` → `-category:promotions` を除去してプロモーションも含める
- `--include-social` → `-category:social` を除去してソーシャル通知も含める

**自動除外されるメール:**
1. **プロモーション（CATEGORY_PROMOTIONS）** — 広告・セール案内・ウェビナー招待など、Gmailが自動分類したプロモーションメールは処理対象外。確認も不要。
2. **ソーシャル（CATEGORY_SOCIAL）** — Facebook・Twitter・LinkedInなどSNSからの通知メールは処理対象外。確認も不要。
3. **保留ラベル（_hold/today）** — 今日は処理しないと判断したメール。翌日以降に再処理。

**対象となるメール:**
- **メイン（CATEGORY_PERSONAL）** — 個人からのメール、ビジネスメールなど返信が必要な可能性があるもの
- **更新情報（CATEGORY_UPDATES）** — 請求書・注文確認など、確認が必要だが返信不要なことが多いもの

**保留メールについて:**
- `_hold/today` ラベルが付いたメールは通常の処理から除外される
- このラベルは毎日0時に自動解除される想定（n8nワークフローで実装）
- `--include-hold` オプションで強制的に処理対象に含めることも可能

### 3. 返信要否の自動分類（3段階）

各メールのヘッダー・本文を分析し、**3段階（返信必要・保留・返信不要）で自動判定**する。

#### 3.1 分類カテゴリ

| カテゴリ | 分類 | 判定基準 |
|---------|------|---------|
| 📧 **個人からの問い合わせ** | ✅ 今すぐ返信 | 顧客・取引先からの質問、見積依頼、相談など |
| 📅 **日程調整依頼** | ✅ 今すぐ返信 | 打ち合わせ・面談の日程確認 |
| 💼 **ビジネス提案** | ⏸️ 保留 | 協業提案、サービス導入相談など（要検討） |
| 🔍 **要調査・検討** | ⏸️ 保留 | 回答に調査や確認が必要なもの |
| ⏰ **後日対応でOK** | ⏸️ 保留 | 緊急性が低い、期限に余裕があるもの |
| 🤖 **自動通知** | ❌ 不要 | システム通知、アラート、リマインダー |
| 📣 **広告・プロモーション** | ❌ 不要 | キャンペーン、セール案内、お買い物情報 |
| 📰 **メールマガジン** | ❌ 不要 | ニュースレター、定期配信 |
| 📢 **イベント案内** | ❌ 不要 | セミナー案内、ウェビナー招待（一方向） |
| 📊 **社内システム通知** | ❌ 不要 | SNS広告リード通知、Google Meet メモ等 |

#### 3.1.1 保留の判断基準

以下のいずれかに該当する場合、保留と判定:

**緊急性の判断:**
- 期限・締切が明記されていて、まだ余裕がある
- 「お時間のあるときに」「ご都合の良い時に」などの表現がある
- 質問内容が複雑で、調査・確認が必要

**検討が必要なケース:**
- 新規取引・協業の提案（社内判断が必要）
- 価格交渉・条件変更の依頼
- 技術的な確認が必要な問い合わせ

**保留にしない（今すぐ対応）ケース:**
- 「至急」「緊急」「本日中に」などの表現がある
- 既存顧客からの問い合わせ
- 日程調整で候補日に今日・明日が含まれる
- クレーム・トラブル対応

#### 3.2 返信不要の判定ルール

以下のいずれかに該当する場合、返信不要と判定:

**送信者アドレスのパターン:**
- `noreply@`, `no-reply@`, `donotreply@`
- `news@`, `newsletter@`, `info@`（大量配信系）
- `mailer-daemon@`, `postmaster@`
- `notification@`, `notify@`, `alert@`

**件名のパターン:**
- 「お買い物」「キャンペーン」「セール」「割引」「ポイント」
- 「メールマガジン」「ニュースレター」「配信」
- 「【PR】」「【広告】」「【お知らせ】」（一方向案内）
- 「通知」「notification」「alert」「reminder」
- 「会議メモ」「Meeting notes」

**本文のパターン:**
- 配信停止・購読解除リンクがある
- 「このメールは自動送信」「This is an automated message」
- 「返信不可」「このメールには返信できません」

**送信者ドメインのパターン（既知の広告・通知系）:**
- `*.rakuten.co.jp`（楽天系プロモーション）
- `*.amazon.co.jp`（Amazon系通知）
- `*.google.com`（Google自動通知）
- `*.chatwork.com`（Chatwork通知）
- `satofull.jp`, `furusato-tax.jp`（ふるさと納税系）

#### 3.4 分類の修正

ユーザーが「分類を修正」を選択した場合:
- 個別のメールを「今すぐ返信」⇔「保留」⇔「返信不要」に変更可能
- 修正結果は学習データとして保存（将来の分類精度向上）

#### 3.5 保留メールの処理

保留と分類されたメールには `_hold/today` ラベルを付与:

```bash
# _hold/today ラベルがなければ作成
gws gmail users labels create --params '{"userId": "me"}' --json '{"name": "_hold/today"}'

# 保留ラベルを付与（受信トレイには残す）
gws gmail users messages modify --params '{"userId": "me", "id": "MESSAGE_ID"}' --json '{
  "addLabelIds": ["HOLD_LABEL_ID"]
}'
```

**保留の仕組み:**
- `_hold/today` ラベルが付いたメールは、その日の `/reply-mail` では処理されない
- 翌日以降、このラベルが解除されると再び処理対象になる
- ラベル解除は n8n ワークフロー（毎日0時に自動実行）で行う

### 5. 保留メールの処理

保留と分類されたメールに `_hold/today` ラベルを付与:

```bash
# _hold/today ラベルのIDを取得（なければ作成）
gws gmail users labels list --params '{"userId": "me"}' --fields "labels(id,name)"

# ラベルがなければ作成
gws gmail users labels create --params '{"userId": "me"}' --json '{"name": "_hold/today"}'

# 保留ラベルを付与（受信トレイには残す）
gws gmail users messages modify --params '{"userId": "me", "id": "MESSAGE_ID"}' --json '{
  "addLabelIds": ["HOLD_LABEL_ID"]
}'
```

**保留メールの翌日処理:**
- n8n ワークフローで毎日0時に `_hold/today` ラベルを解除
- 翌日の `/reply-mail` で再び処理対象になる
- 緊急対応が必要になった場合は `--include-hold` オプションで即時処理可能

#### 6.2 ユーザー確認の選択肢

各メールに対して以下の選択が可能:

| 選択 | 動作 |
|------|------|
| **OK** | 提案されたラベルを適用し、受信トレイから除外 |
| **変更** | 別のラベルを指定してから適用 |
| **受信トレイに残す** | ラベルは付与するが受信トレイにも残す |
| **スキップ** | このメールは何もしない（受信トレイに残る） |

#### 6.3 一括操作

- **「全部OK」**: 表示されているすべての提案を一括承認
- **「全部スキップ」**: すべてのメールを処理せず受信トレイに残す

#### 6.4 振り分け先ラベルの候補

| カテゴリ | 振り分け先ラベル |
|---------|---------------|
| 📣 広告・プロモーション | `_auto/広告` |
| 📰 メールマガジン | `_auto/メルマガ` |
| 🤖 自動通知 | `_auto/通知` |
| 📊 社内システム通知 | `_auto/システム通知` |
| 📢 イベント案内 | `_auto/イベント案内` |
| 📚 研修関連 | `研修関係/{サブラベル}` |

#### 7.8 送信実行

**重要: 必ず元のスレッドに返信（リメール）形式で送信すること**

新規メールとして送信してはいけない。必ず `threadId` を指定してスレッドに返信する:

```bash
gws gmail users messages send --params '{"userId": "me", "threadId": "THREAD_ID"}' --json '{
  "raw": "BASE64_ENCODED_REPLY"
}'
```

**理由:**
- スレッドの文脈が維持される
- 相手が過去のやり取りを参照できる
- メールの追跡・管理がしやすい

#### 7.9 送信後のラベル振り分け（必須）

1. 学習履歴を確認
2. ラベルを提案
3. ユーザー確認後、ラベル適用
4. 受信トレイから除外（アーカイブ）

## 保留メールの運用

### 自動解除の仕組み
- `_hold/today` ラベルは毎日0時に n8n ワークフローで自動解除
- 解除されると翌日の `/reply-mail` で再び処理対象になる

### 保留メールを即時処理したい場合
```
/reply-mail --include-hold           # 保留メールも処理対象に含める
```

### 保留メールの確認
```
/reply-mail --label=_hold/today --list   # 現在保留中のメール一覧
```

## メール送信の基本ルール（最重要）

**すべてのメール送信は、必ず元のスレッドに返信（リメール）形式で行うこと。**

新規メールとして送信してはいけない。必ず `threadId` を指定する。

**理由:**
- スレッドの文脈が維持される
- 相手が過去のやり取りを参照できる
- メールの追跡・管理がしやすい
- 重複メールと誤解されにくい

## 配信エラー時の対応

**配信エラー（Delivery Status Notification Failure）が発生した場合:**

### 1. 正しいメールアドレスの特定

配信エラーになった宛先アドレスが間違っている可能性が高い。元のスレッドまたは過去のやり取りから正しいアドレスを検索する:

```bash
# 送信者名で過去のメールを検索
gws gmail users messages list --params '{"userId": "me", "q": "from:相手の名前またはドメイン", "maxResults": 10}'
```

### 2. 正しいアドレスで再送信（リメール形式）

元のスレッドIDを指定して返信形式で再送信:

```bash
gws gmail users messages send --params '{"userId": "me", "threadId": "ORIGINAL_THREAD_ID"}' --json '{
  "raw": "BASE64_ENCODED_REPLY"
}'
```

### 3. 配信エラー通知の処理

元のエラー通知メールはゴミ箱に移動する:

```bash
gws gmail users messages trash --params '{"userId": "me", "id": "ERROR_MESSAGE_ID"}'
```

## エラー対応

- **Gmail APIエラー**: `gws auth status` で認証状態を確認
- **CRM接続エラー**: lts-sales-crm の DATABASE_URL を確認
- **ラベルが見つからない**: `gws gmail users labels list` でラベルID確認
