# 04 開発と自動化のフロー — AI Development Flow

> **非エンジニアが安全かつ高速にシステムを構築・運用するための標準プロセス。**

---

## ⚠️ 安全装置について

誤って **Gドライブ（Google Drive）上** で Claude Code を起動すると、システム保護のため警告が表示されます。
**必ずDドライブのプロジェクトフォルダ内で作業してください。**「PCが壊れた！」とパニックにならず、落ち着いてDドライブに移動してから再起動してください。

```
✅ 正しい起動場所の例:
   cd D:/scripts/ryouchiku-systems
   claude
```

## 1. 開発フローの全体像

```
[1. 要件定義]  →  [2. Plan Mode]  →  [3. 人間がOK]  →  [4. 実行・テスト]  →  [5. 本番反映]  →  [6. ナレッジ化]
```

---

## 2. ステップ詳細

### Step 1: 要件定義
AIに作業を依頼する前に、以下を自分の言葉で整理する。

- **何のために作るのか？**（目的）
- **誰が使うのか？**（対象者）
- **どんな動作をすれば成功か？**（完了条件）
- **やってはいけないことは何か？**（制約）

> ヒント: 完璧な要件でなくてよい。まずAIに「こういうことをしたい」と相談するところから始めてもよい。

---

### Step 2: Plan Mode（計画の提示）【必須】

AIに対して **「Plan Mode」** として計画を出させること。

**Plan Modeとは:**
- AIが「何を、どの順番で、どのように変更するか」を**実行前に提示**するモード。
- コードの変更、ファイルの作成・削除、外部APIの呼び出しなど、**影響範囲が大きい操作の前に必ず実施**する。

**使い方:**
```
Claude Code では `/plan` コマンド、または会話で「まず計画だけ教えて」と伝える。
```

**Plan Modeをスキップしてよいケース:**
- 既存ファイルの軽微な文言修正
- 新規ファイルの追加（既存への影響がない場合）
- ユーザーが明示的に「Plan Modeをスキップして」と指示した場合

---

### Step 3: 人間がOKを出す

AIが提示した計画を確認し、問題なければ「OK」「進めて」と伝える。

**確認すべきポイント:**
- [ ] 変更・削除されるファイルに重要なデータが含まれていないか
- [ ] 想定外のファイルや設定が変更されていないか
- [ ] 本番環境に直接影響しないか（テスト環境での実施か）

---

### Step 4: 実行・テスト

AIが計画に沿って実装を行う。スタッフは以下を担当する。

- テスト環境での動作確認
- 期待通りの結果が出ているかのチェック
- エラーが出た場合はAIに状況を共有して対処を依頼

---

### Step 5: 本番反映（デプロイ）

**LTSではスピードを最優先するため、作成したスタッフ本人が本番環境へのデプロイを行ってよい。**
承認フローで時間を無駄にするより、素早く価値を届けることを優先する。

**ただし、本番反映前に必ず確認する:**
- [ ] テスト環境で期待通りに動作したか
- [ ] 本番データへの影響範囲を把握しているか
- [ ] ロールバック手順を把握しているか（万が一に備えて）

AIは「本番環境に反映してよいか？」をユーザーに確認してから、デプロイ作業をサポートする。

**「本番反映（デプロイ）」とは何か？— ツール別の具体例**

| ツール | 本番反映の意味 | 具体的な手順 |
|--------|--------------|-------------|
| **n8n** | テスト完了したJSONを本番のn8n環境にインポートし、ワークフローを「Active」にすること | n8n画面でJSONをインポート → ワークフローを開く → 右上のトグルを「Active」にする |
| **GAS** | GASエディタで「新しいデプロイ」を作成し、最新のコードをWebアプリ等に反映すること | GASエディタ → デプロイ → 新しいデプロイ → 種類を選択して「デプロイ」ボタンを押す |

---

### Step 6: ナレッジ化（lts-knowledge の更新）

新しい知見・解決策・ベストプラクティスを得たら、`lts-knowledge` リポジトリを更新する。

**更新フロー:**
1. 担当スタッフが該当のMarkdownファイルを編集し、Pull Requestを作成。
2. 管理職（または担当チームリーダー）がレビューし、承認・マージ。
3. 全社に共有（必要に応じてGoogle chatなどで周知）。

> **AIからの自動提案:** AIが新しい有益な解決策を見つけた際は、`lts-knowledge` へのPull Request作成を自発的に提案します。

---

## 3. ツール別の注意事項

### n8nワークフロー
- 本番ワークフローを直接編集しない。必ず複製して作業する。
- JSONはGitで管理し（`D:/scripts/{client-id}/n8n/`）、変更履歴を残す。

### 外部API連携
- 本番APIキーを使ったテストは最小限にとどめる。
- テスト用サンドボックスが用意されているサービスは必ずサンドボックスを使う。

### データベース操作
- `DELETE`, `DROP`, `TRUNCATE` などの操作は、必ずバックアップを確認してから実行。
- AIにデータベース操作を依頼する際は、**Plan Modeを必ず使う**。

---

## 4. トラブル時の基本姿勢

- **同じ操作を繰り返さない** — エラーが出たら、原因を理解してから次の手を考える。
- **AIを責めない、諦めない** — アプローチを変えれば解決することがほとんど。
- **記録する** — 解決した問題は `lts-knowledge` に残し、次の人（と次のAI）を助ける。
