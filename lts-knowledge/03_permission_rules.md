# 03 権限とセキュリティ — Permission Rules

> **最小権限の原則に基づき、必要な人だけが必要な情報にアクセスできる状態を保つ。**

---

## 1. 権限の4階層

| レベル | 名称 | 対象者 | 含まれる情報の例 |
|--------|------|--------|-----------------|
| **① 全社公開** | General | 全スタッフ・全AI | ナレッジベース（`lts-knowledge`）、全体用ツールの使い方、会社の考え方 |
| **② チーム限定** | Project | 特定のプロジェクトチームのみ | 個別案件のソースコード、顧客データ、案件固有のn8nワークフロー |
| **③ 経営陣限定** | Executive | 経営陣・役員のみ | 財務データ、人事情報、経営ダッシュボード、KPIレポート |
| **④ 代表限定** | Confidential | 龍竹（代表）と指定された特定の人のみ | 最高機密の意思決定資料、M&A関連、役員報酬 |

---

## 2. Gitリポジトリの権限管理

各リポジトリの可視性（Visibility）と招待メンバーは、上記のレベルに対応して設定する。

```
① lts-knowledge         → Public または Organization全体に公開
② {client-id}           → 担当チームのメンバーのみ招待（Private）
③ lts-executive         → 経営陣のみ（Private、厳格なアクセス制限）
④ lts-confidential      → 代表 + 指定者のみ（Private）
```

**ルール:**
- リポジトリを作成する際は、最初に適切な権限レベルを決定してから作成する。
- 後から権限を緩める（例: PrivateをPublicに変更）場合は、必ず龍竹（代表）の承認を得る。
- 権限を厳しくする（例: PublicをPrivateに変更）場合はチームリーダーの判断で実施可。

---

## 3. Driveの権限管理

Driveの共有フォルダも、同じ4階層に従って共有範囲を設定する。

```
01_社内管理/             → ① 全社共有
02_自社事業(LTS)/        → ① 全社共有（または ② 担当チーム）
03_グループ会社/         → ② 関係者のみ
04_clients/{client-id}/  → ② 担当チームのみ
```

**ルール:**
- 「リンクを知っている全員が閲覧可能」設定は原則禁止。必ず特定の人・グループに共有する。
- クライアントに共有するフォルダは、社内専用のファイルと必ず分離する。

---

## 4. APIキー・シークレットの管理

| 種類 | 保管場所 | 禁止事項 |
|------|----------|----------|
| 本番のAPIキー | 各サービスの環境変数 / シークレットマネージャー | Gitにコミット禁止、チャットに貼り付け禁止 |
| 開発・テスト用キー | ローカルの `.env` ファイル（`.gitignore` 必須） | 本番サービスと同じキーを使い回すことは禁止 |
| 共有が必要な場合 | Drive ③経営陣限定 フォルダ内の暗号化ファイル | メールやSlackでの平文送信禁止 |

---

## 5. インシデント発生時の対応

1. **即座に報告** — 龍竹（代表）と担当チームリーダーに連絡。
2. **アクセス遮断** — 漏洩が疑われるキーを即座に無効化・再発行。
3. **記録** — 何が起きたか、どう対処したかを `lts-knowledge` に追記（Pull Request経由）。
4. **再発防止** — 権限設定を見直し、必要であれば本ドキュメントを更新。
