# 05 クライアントリポジトリ 命名規則と運用ルール

> **「リポジトリを作る前にルールを決める」— これが技術的負債を生まない唯一の方法。**

---

## 1. 命名規則

### 1-1. クライアント ID（最重要・全ての基準）

クライアントIDは **Git リポジトリ名・Drive フォルダ名・n8n ワークフロー名** に共通して使う「識別子」です。

**形式:** `半角英小文字` + `ハイフン（-）` のみ

```
OK: sky-connect, yamada-ltd, abc-trading
NG: SkyConnect, sky_connect, SKY, スカイ, skyConnect
```

**決め方の優先順位:**
1. 英語社名の略称（例: `sky-connect` ← Sky Connect 株式会社）
2. 読み仮名のローマ字（例: `yamada-shoji` ← 山田商事）
3. 略称が衝突する場合は末尾に業種を付ける（例: `yamada-food`）

> 一度決めたIDは **変更しない**。変えると Git・Drive・n8n・CLAUDE.md の全箇所を修正する羽目になる。

---

### 1-2. リポジトリ名のパターン

クライアント1社につき、作業の種類・量に応じて以下3パターンから選ぶ。

| パターン | リポジトリ名 | 使うケース |
|---------|------------|-----------|
| **A. 統合型**（推奨） | `{client-id}` | 1社の全作業を1リポジトリに収める（小〜中規模） |
| **B. サービス分割型** | `{client-id}-{service}` | 作業が独立して大きい場合（例: CRM と Web が別チーム） |
| **C. 禁止** | `client-{client-id}` | `client-` プレフィックスは**使わない**（冗長で意味がない） |

**パターン B のサービス名の例:**

| サービス名 | 意味 |
|-----------|------|
| `crm` | 顧客管理・営業自動化 |
| `web` | Webサイト・LP |
| `n8n` | n8n ワークフロー集 |
| `app` | アプリ（Next.js 等） |
| `auto` | スクレイピング・自動化スクリプト |

**例:**
```
sky-connect              ← パターン A（全作業を1リポジトリに）
yamada-food-crm          ← パターン B（CRM だけ独立）
yamada-food-web          ← パターン B（Web だけ独立）
```

> **判断基準:** 迷ったらパターン A から始める。1つのリポジトリが 5,000行・10ファイルを超えたら分割を検討する。

---

### 1-3. Drive フォルダとの対応

Git と Drive は **クライアントID で必ず一致** させる。

| Git（Dドライブ） | Drive（Gドライブ） |
|----------------|------------------|
| `D:/scripts/sky-connect/` | `G:/.../04_clients/sky-connect_Sky Connect株式会社/` |
| `D:/scripts/yamada-food-crm/` | `G:/.../04_clients/yamada-food_山田フード株式会社/` |

> Drive フォルダ名は `{client-id}_{正式会社名}` の形式（`02_folder_structure.md` のルールを継承）。
> サービス分割の場合も Drive フォルダは **1社1フォルダ**（サービスはサブフォルダで管理）。

## 2. リポジトリ作成タイミング

### 作成する（OK）条件 — **全て満たすこと**

| 条件 | 確認方法 |
|------|---------|
| ✅ 契約が成立している（口頭・書面どちらでも） | 龍竹から「進めて」の確認 |
| ✅ 実際にコードや自動化スクリプトを書く作業が発生する | 提案書・見積もりに開発作業が含まれる |
| ✅ クライアントIDが確定している | `05_client_repo_rules.md` のルールに基づき命名済み |

### 作成しない（NG）条件

| 状況 | 理由 |
|------|------|
| ❌ 商談中・提案書作成のみ | 受注しなければ不要なリポジトリが残る |
| ❌ メールや資料の共有だけ | Drive で十分（コードがない） |
| ❌ 「とりあえず」「後で使うかも」 | 迷ったら作らない |

### 作成の手順（Claude Code が代行可）

```bash
# 1. ローカルフォルダ作成 & Git 初期化
mkdir D:/scripts/{client-id}
cd D:/scripts/{client-id}
git init

# 2. 最小ファイルを作成
# - CLAUDE.md（このファイルのテンプレートを使う）
# - README.md
# - .gitignore

# 3. GitHub に Private リポジトリを作成してプッシュ
gh repo create kazui0907/{client-id} --private --source=. --push
```

## 3. リポジトリ内の標準構成

新規クライアントリポジトリには以下の構成を **最初から用意する**（空でもファイルを置く）。

```
{client-id}/
├── CLAUDE.md          ← AIへの指示書（必須・後述のテンプレートを使う）
├── README.md          ← 概要・セットアップ方法（人間向け）
├── .gitignore         ← .env, node_modules 等を除外
├── n8n/               ← n8n ワークフロー JSON（使う場合）
├── scripts/           ← 自動化スクリプト（Python / GAS / JS）
└── docs/              ← 設計メモ・API仕様・ミーティング議事録
```

## 4. CLAUDE.md テンプレート（クライアント用）

新しいクライアントリポジトリを作ったら、以下をコピーして `CLAUDE.md` を作成する。

```markdown
# {正式クライアント名} プロジェクト ルールブック

## クライアント基本情報

| 項目 | 内容 |
|------|------|
| 会社名 | {正式会社名} |
| クライアントID | {client-id} |
| 担当者 | {担当者名} / {連絡先} |
| 契約開始 | {YYYY-MM-DD} |
| 主な業種 | {業種} |

## プロジェクト概要

{このクライアントに対して何を提供しているかを2〜3行で記載}

## 作業内容

- [ ] {作業1}
- [ ] {作業2}

## 技術スタック

| 用途 | 技術 |
|------|------|
| 自動化 | n8n |
| スクリプト | Python / GAS |
| データ | Google Sheets |

## 重要なリンク・ID

| 種類 | 値 |
|------|-----|
| スプレッドシート | {Google Sheets URL または ID} |
| n8n ワークフロー | {ワークフロー名} |
| Drive フォルダ | {Google Drive フォルダ URL} |

## 注意事項・制約

- {このクライアント特有のルールや禁止事項を記載}

{本番反映の具体的な手順を記載}
```

## 5. 旧 clients/ フォルダからの移行ルール（22件）

モノレポ解体で旧 `clients/` 以下にあった22件を独立リポジトリ化する際の手順。

### 移行の優先順位

| 優先度 | 対象 | 基準 |
|-------|------|------|
| 🔴 高 | 稼働中のシステムがある | n8n ワークフローが Active、定期スクリプトが動いている |
| 🟡 中 | 今後作業予定がある | 3ヶ月以内に開発・修正の予定がある |
| 🟢 低 | 休眠・完了済み | 作業が完全に終了、今後の予定なし |

### 移行手順

```bash
# 旧 clients/{client-id} から新リポジトリに移行
cd D:/scripts
cp -r {旧パス}/{client-id} ./{client-id}
cd {client-id}
git init
# CLAUDE.md・.gitignore を追加
gh repo create kazui0907/{client-id} --private --source=. --push
```

### 移行後の確認チェックリスト

- [ ] CLAUDE.md が作成されている
- [ ] .gitignore で .env が除外されている
- [ ] GitHub に Private リポジトリとして登録されている
- [ ] Drive フォルダ名が `{client-id}_{正式会社名}` になっている
- [ ] `02_folder_structure.md` の「現在のGitHubリポジトリ一覧」に追記した

## 6. よくある間違いと正しい対応

| NG パターン | 正しい対応 |
|-----------|-----------|
| `client-sky-connect` という名前にした | `sky-connect` に修正（`client-` プレフィックス不要） |
| 商談中なのにリポジトリを作った | 削除する。受注してから作る |
| CLAUDE.md を書かずに作業を始めた | 最初に CLAUDE.md を作ってから作業開始 |
| 1社に10個リポジトリを作った | 原則1社1リポジトリ。5,000行超えたら分割を検討 |
| リポジトリ名に日本語を使った | 英数字とハイフンのみに変更 |
